# 求职助手平台 - 任务拆分与实现方案

## 项目总览

一个全流程求职辅助平台，从岗位信息收集、简历定制投递、试题搜索、到模拟面试，覆盖求职全链路。

### 用户系统

- [ ] 注册/登录（邮箱 + 密码，或 OAuth 第三方登录）
- [ ] 用户数据隔离（每个用户的简历、投递记录、面试记录互不可见）
- [ ] 数据安全：简历是高度敏感的个人信息，需加密存储、传输层 HTTPS、API 鉴权
- [ ] 用量控制：LLM API 按 token 计费，需做每用户每日用量限额

## 模块一：岗位信息收集

### 1.1 BOSS直聘数据采集（已有基础）

**现状**：已有 `scrape.py` 可批量抓取搜索结果的职位详情，输出 Excel，包含 13 个字段。

**待优化**：
- [ ] 支持多城市、多关键词批量搜索
- [ ] 增量采集：记录已抓取 URL，避免重复
- [ ] 定时采集：cron 定时任务，每日自动抓取新岗位
- [ ] 反爬对抗：随机延时、User-Agent 轮换、Cookie 管理
- [ ] 错误重试：网络异常时自动重试，记录失败 URL

### 1.2 数据存储

- [ ] **SQLite / PostgreSQL 数据库**：结构化存储所有岗位数据
  - 表设计：`jobs`（岗位主表）、`companies`（公司表）、`hr_contacts`（HR 信息表）
  - 字段对齐 scrape.py 已有的 13 个字段
- [ ] **去重策略**：以 URL 或 岗位ID 为唯一键
- [ ] **全文检索**：岗位描述、公司介绍等长文本字段建全文索引，方便后续搜索

### 1.3 数据清洗与归一化（重要）

不同平台、不同 HR 写的薪资格式千差万别，入库前必须清洗：

| 原始值 | 清洗后 |
|--------|--------|
| `15k-25k` | `15k-25k`，月薪 |
| `15K-25K * 15薪` | `15k-25k`，月薪，15薪 |
| `15000-25000` | `15k-25k` |
| `面议` | `面议`（标记为缺失，薪资分析时排除） |
| `300-500/天` | 实习/日薪岗，单独标记 |

- [ ] 薪资字段正则提取 + 统一为月薪数值（方便后续比较和筛选）
- [ ] 经验要求统一：「1-3年」「1年以下」「应届生」「不限」
- [ ] 学历要求统一：「大专」「本科」「硕士」「博士」「不限」
- [ ] 工作地点拆分为 省/市/区 三级

### 1.4 多平台扩展

- [ ] 猎聘、智联招聘、拉勾等平台的采集适配器
- [ ] 统一数据格式，各平台数据归一化到同一 schema

> ⚠️ **法律风险提示**：大规模、高频抓取招聘网站数据可能违反其服务条款（ToS）。建议控制采集频率（每日 1-2 次）、仅采集与个人求职相关的岗位、不公开传播数据。如果面向多用户提供服务，需额外注意合规。

### 实现方案

```
技术栈：Python + Playwright（比 Selenium 反检测能力更强）+ PostgreSQL

文件结构：
  collectors/
    base.py           # 采集器基类（统一接口：search → fetch_detail → normalize）
    boss_zhipin.py    # BOSS直聘（重构现有 scrape.py）
    liepin.py         # 猎聘（预留）
  storage/
    db.py             # 数据库操作
    models.py         # ORM 模型
    cleaner.py        # 数据清洗 & 归一化
  scheduler/
    cron_jobs.py      # 定时任务

## 模块二：个性化简历修改与投递

### 2.1 简历解析

- [ ] 支持用户上传 PDF / Word / Markdown / 纯文本 / 图片格式的基础简历
- [ ] 解析为结构化数据：个人信息、教育经历、工作经历、项目经历、技能标签、自我评价
- [ ] **解析难点提醒**：PDF 是最常见格式，也是最难解析的——
  - 双栏排版易导致文字顺序错乱
  - 特殊中文字体编码可能导致乱码
  - 表格/图表中的内容难以提取
  - 图片型 PDF 需 OCR
  - 建议：优先引导用户上传 Markdown 或粘贴纯文本简历；PDF 解析用 PyMuPDF + 规则纠错，不追求 100% 准确，解析结果始终允许用户手动修正
- [ ] 解析后存入用户简历库，支持多版本管理（原始版 + 各岗位定制版）

### 2.2 岗位-简历匹配分析

- [ ] **硬性要求提取**：从岗位描述中提取技能关键词、经验年限、学历要求
- [ ] **软性要求提取**：从岗位描述 + 公司介绍中提取文化特征、团队风格
- [ ] **匹配度打分**：简历与岗位的匹配程度量化（硬性要求 + 软性要求加权）
- [ ] **差异分析**：列出简历缺少的岗位要求，高亮关键差距
- [ ] **ATS 兼容检查**：大厂普遍用 ATS（Applicant Tracking System）初筛简历——
  - 检测简历是否会被 ATS 正确解析（无图片文字、无复杂表格、标准章节标题）
  - 检测关键词密度是否达到 ATS 通过阈值

### 2.3 AI 简历改写 + 求职信生成

- [ ] **简历改写**（调用 LLM API）：
  - 输入：基础简历（结构化 JSON）+ 岗位信息 + 公司文化特征 + ATS 关键词要求
  - 输出：定制化简历
  - 原则：保留真实经历不造假、调整侧重点和措辞、技能描述对齐 JD 关键词、自我评价融入企业文化
- [ ] **求职信（Cover Letter）生成**：
  - 按「为什么对这个岗位感兴趣 → 我的匹配点 → 为什么选择这家公司 → 期待面试」结构生成
  - 引用公司业务/产品/文化细节（从公司介绍字段提取），显得做了功课
- [ ] 生成后允许人工审阅修改
- [ ] 导出 PDF：
  - 中文 PDF 生成注意：WeasyPrint 对中文支持有坑，推荐方案是用 HTML 模板 + Playwright 渲染打印为 PDF
  - 或使用 LaTeX 模板（中文支持最好的是 XeTeX + ctex 宏包）

### 2.4 简历投递

> ⚠️ **风险提示**：自动投递是最高风险环节。BOSS直聘等平台有严格的反自动化检测（行为模式分析、验证码），账号可能被封禁。

- [ ] **方案 A（低风险，推荐 MVP）**：平台内沟通自动化——
  - 自动发送打招呼消息（预设话术模板 + 岗位相关定制）
  - 不自动发附件简历，由用户手动确认后发送
  - 记录沟通状态
- [ ] **方案 B（高风险）**：全自动投递——
  - Playwright 模拟点击 + 上传简历
  - 需配合行为随机化（鼠标轨迹模拟、随机停顿、打字速度变化）
  - 每日投递数量限制避免触发风控
- [ ] 投递记录管理：记录投递时间、岗位、简历版本、状态（已投/已读/回复/面试/录用/拒绝）
- [ ] 投递漏斗分析：可视化投递转化率

### 实现方案

```
技术栈：Python + LLM API（Claude/GPT）+ Playwright（PDF 渲染 + 投递自动化）

文件结构：
  resume/
    parser.py          # 简历解析（PDF/Word/Markdown → JSON）
    matcher.py         # 岗位-简历匹配分析 + ATS 检查
    rewriter.py        # AI 简历改写 + 求职信生成
    templates/         # HTML/LaTeX 简历模板
    pdf_renderer.py    # PDF 渲染（Playwright PDF 方案）
  delivery/
    boss_deliver.py    # BOSS直聘沟通/投递
    tracker.py         # 投递状态追踪

## 模块三：笔面试试题搜索

### 3.1 试题来源

| 来源 | 内容类型 | 采集难度 | 采集方式 |
|------|---------|---------|---------|
| 牛客网 | 笔试真题、面经 | ⭐⭐⭐ | 页面抓取，需登录，有反爬 |
| 小红书 | 面经分享 | ⭐⭐⭐⭐⭐ | 反爬极强（App 端加密），优先考虑搜狗微信搜索间接获取 |
| 掘金 | 技术面试题 | ⭐⭐ | 页面抓取，反爬较轻 |
| 知乎 | 公司面经、「XX公司面试体验」 | ⭐⭐⭐ | 页面抓取 |
| LeetCode / 力扣 | 算法题 | ⭐ | 有公开 API / 社区整理的题目列表 |
| GitHub | 面试题合集仓库 | ⭐ | GitHub API，搜索 `awesome-interview`、`CS-Notes`、`JavaGuide` 等知名仓库 |
| 看准网/脉脉 | 公司面经、员工评价 | ⭐⭐⭐ | 需登录，页面抓取 |

> 建议优先级：GitHub 仓库（零成本 + 高质量）→ LeetCode API → 掘金 → 牛客网 → 知乎 → 小红书（最后，难度大）

### 3.2 数据获取策略

除了自己爬，还可以利用已有资源：
- [ ] GitHub 上的面试题合集仓库（如 CS-Notes、JavaGuide、前端面试宝典等），直接 clone 解析 Markdown 文件入题库
- [ ] LeetCode 社区整理的「高频题」列表
- [ ] BOSS直聘岗位描述里的「常见面试题」标签（个别公司会写）

### 3.3 功能设计

- [ ] **按岗位/公司搜索**：输入岗位名称 + 公司名，返回相关试题
- [ ] **按技能标签搜索**：如 "Python + 机器学习 + 面试题"
- [ ] **语义搜索**：用 Embedding 向量做语义匹配——
  - 用户搜「Python 并发编程」，能匹配到「Python GIL」「多线程」「协程 asyncio」等相关题
  - 技术方案：text-embedding-3-small 或 bge-large-zh 生成向量 → pgvector / Chroma 存储 → 余弦相似度检索
- [ ] **试题分类**：算法题、系统设计、行为面试、技术基础（八股文）、项目经历深挖
- [ ] **试题详情页**：题目 + 参考答案 + 来源链接 + 出现频次 + 相关岗位
- [ ] **收藏/标记**：已掌握、需复习、重点
- [ ] **导出**：将筛选的试题导出为 PDF/打印版，方便线下刷题

### 3.4 数据存储

- [ ] 试题库表：题目、答案、标签、来源、公司、岗位、出现频次、embedding 向量
- [ ] 面经表：原文链接、摘要、公司、岗位、发布时间
- [ ] 用户标记表：用户对试题的掌握状态

### 实现方案

```
技术栈：Python + Playwright/httpx + BeautifulSoup + PostgreSQL + pgvector（向量检索）

文件结构：
  questions/
    sources/
      github_repos.py    # GitHub 仓库解析（优先）
      leetcode.py        # LeetCode/力扣
      juejin.py          # 掘金
      nowcoder.py        # 牛客网
      zhihu.py           # 知乎
    search.py            # 统一搜索接口（关键词 + 语义搜索）
    embedder.py          # Embedding 生成 & 向量检索
    db.py                # 试题库存储
```

## 模块四：AI 模拟面试

> 分两步走：**Phase 1** 纯语音交互快速跑通全流程 → **Phase 2** 升级为数字人视频面试。

### 4.1 整体架构

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│  前端界面    │ ←→ │  后端服务     │ ←→ │  AI 服务层   │
│ (Next.js)   │    │ (FastAPI)    │    │             │
│             │    │             │    │ - LLM 出题   │
│ - 语音交互 UI │    │ - 会话管理   │    │ - STT 语音识别│
│ - 数字人渲染  │    │ - 评分引擎   │    │ - TTS 语音合成│
│ - 视频录制   │    │ - 报告生成   │    │ - 数字人驱动  │
│ - 面试回放   │    │             │    │             │
└─────────────┘    └──────────────┘    └─────────────┘
```

### 4.2 公共部分（两阶段共享）

#### 4.2.1 面试前：面试配置

- [ ] **选择岗位**：从岗位库中选择目标岗位，或手动粘贴岗位 JD 链接/文本
- [ ] **面试类型**：技术面 / HR 面 / 综合面 / 行为面试
- [ ] **面试模式**：
  - **练习模式**：可暂停、每题不限时、答完可看参考要点、可重答同一题
  - **考试模式**：严格计时、不可暂停、不可回退，模拟真实面试压力
- [ ] **难度等级**：初级 / 中级 / 高级
- [ ] **面试时长**：15分钟 / 30分钟 / 45分钟 / 60分钟
- [ ] **考察维度权重配置**：技术能力、沟通表达、项目经验、问题解决、文化匹配

#### 4.2.2 面试流程引擎（共享）

LLM 扮演面试官角色，整个面试流程由 LLM 驱动：

```
开场白 → 自我介绍 → 技术提问 → 追问（根据回答动态深入）→ 行为面试 → 候选人提问 → 结束语
```

- [ ] 每道题限时回答，倒计时结束自动进入下一题
- [ ] 追问策略：LLM 判断候选人回答是否足够深入，不够则追问（最多 2 层追问）
- [ ] 出题策略：
  - 第一轮：从试题库随机抽取匹配岗位的题目
  - 追问轮：LLM 根据候选人回答实时生成追问
  - 穿插行为面试题（"说说你遇到过的最大技术挑战"）
- [ ] **边界情况处理**（LLM prompt 中需明确指令）：
  - 候选人说「不知道」→ 给简要提示（练习模式）或礼貌跳过（考试模式）
  - 候选人回答完全离题 → 委婉引导回正题
  - 候选人沉默超时（30s+）→ 友好提醒并推进到下一题
  - 候选人给出非常简短敷衍的回答 → 追问一次「可以展开说说吗」
- [ ] **上下文窗口管理**（关键）：
  - 45 分钟面试，所有对话历史 + system prompt + 岗位信息可能超过 8K-16K tokens
  - 策略：每 3 轮对话由 LLM 做一次「摘要压缩」作为滚动上下文，替代原始长对话
  - system prompt 中包含岗位信息摘要（非完整 JD），控制在 500 tokens 以内

#### 4.2.3 语音分析（面试中实时采集）

- [ ] **语气词/填充词检测**：统计「嗯、啊、那个、就是说、然后」等词的出现频率，作为表达流畅度指标
- [ ] **语速分析**：实时计算候选人的平均语速（字/秒），与标准语速（汉语正常语速约 3-5 字/秒）对比——
  - 过快 → 可能紧张
  - 过慢 → 可能在组织语言，降低沟通评分
- [ ] **停顿分析**：统计异常长时间停顿（>3 秒）的次数，作为临场反应指标

#### 4.2.4 面试后：评估报告（共享）

每道题记录：
- 面试官提问（文字）
- 候选人回答（语音 → Whisper 转文字）
- 回答耗时、语速、填充词次数
- 追问轮次

LLM 评分引擎逐题分析（每次面试后异步执行）：
- [ ] **技术准确性**（1-10 分）：知识点是否正确、完整、深入
- [ ] **表达清晰度**（1-10 分）：逻辑是否清晰、语言是否流畅
- [ ] **深度与广度**（1-10 分）：是否展示出对相关领域的深入理解
- [ ] **每道题的优势**：具体哪里回答得好
- [ ] **每道题的不足**：遗漏了什么、哪里可以更好
- [ ] **参考回答要点**：提供这道题的高质量回答框架（LLM 生成），供用户对比学习
- [ ] **优化建议**：具体怎么改进、推荐学习资料

综合评估：
- [ ] **总分**（加权计算各维度）
- [ ] **各维度得分雷达图**
- [ ] **语言指标**：平均语速、填充词频率、异常停顿次数
- [ ] **总体面试建议**（200-300 字）
- [ ] **高频问题归纳**：面试中暴露的共性问题，归纳为 1-3 个改进方向
- [ ] **后续学习路线**：针对性提升建议 + 推荐试题

#### 4.2.5 进步追踪（多次面试对比）

- [ ] 记录用户每次面试的总分和各维度得分
- [ ] 生成面试成绩趋势图（折线图），直观看到进步
- [ ] 标注薄弱维度是否在改善（如「表达清晰度」连续 3 次在 5 分以下 → 重点提醒）

报告形式：
- [ ] HTML 网页报告（含雷达图、趋势图等图表，用 ECharts/Chart.js）
- [ ] PDF 导出
- [ ] 面试音频/视频 + 时间轴标注（哪道题、几分几秒开始）

### 4.3 Phase 1：语音交互面试（MVP 先跑通）

纯语音对话，前端 UI 展示面试官头像（静态图片）+ 语音波形动画 + 对话文字流。

#### 交互流程

```
用户点击"开始面试"
  → 后端 LLM 生成开场白文字
  → TTS 转为语音 → 前端播放
  → 用户语音回答 → 前端录音
  → STT 转为文字 → 发给 LLM
  → LLM 判断：继续追问 / 进入下一题 / 结束
  → 循环...
```

#### 前端 UI

- [ ] 面试官静态头像 + 说话时波形/呼吸灯动画（表示"正在说话"）
- [ ] 对话记录实时滚动（类似聊天界面，区分面试官和候选人）
- [ ] 当前题目卡片 + 倒计时
- [ ] 麦克风状态指示（录音中 / 静音 / 等待用户说话）
- [ ] 面试进度条（当前第几题 / 共几题，含时间进度）
- [ ] **面试后**：实时语音分析结果展示（语速、填充词统计等）

#### 技术选型（Phase 1）

| 组件 | 方案 | 说明 |
|------|------|------|
| TTS | Edge TTS（免费） | `edge-tts` Python 库，中文效果不错，零成本 |
| STT | Whisper API 或本地 small 模型 | 本地运行需 ~2GB 显存，CPU 也可跑只是慢 |
| LLM | Claude API / GPT-4o | 出题 + 追问判断 + 评分 |
| 后端实时通信 | FastAPI + WebSocket | LLM 流式输出 → TTS 流式播放，降低首字延迟 |
| 前端录音 | MediaRecorder API | 浏览器原生，WAV 格式发送到后端 |
| 语音分析 | Python 端后处理 | 填充词正则匹配 + 语速计算（STT 时间戳 + 字数） |

#### 延迟优化（关键）

语音面试的核心体验指标是 **首字延迟**（用户说完到听到回复的时间）：

```
用户说完最后一个字
  → [STT: 200-500ms]    Whisper 转文字
  → [LLM: 500-2000ms]   生成回复（流式，取首 token）
  → [TTS: 100-300ms]    合成前几个字即可开始播放
  ─────────────────────
  目标总延迟：< 2 秒
```

- [ ] LLM 流式输出 → TTS 分段合成 → 前端边收边播（不等全部生成完）
- [ ] 预生成开场白（进入面试间时就生成好，用户点开始直接播放）

### 4.4 Phase 2：数字人视频面试（升级）

在 Phase 1 语音交互完全跑通后，将静态面试官替换为有形象、会动、会说话的数字人。

#### 数字人形象方案对比

| 方案 | 效果 | 成本 | 延迟 | 技术门槛 |
|------|------|------|------|---------|
| **A: HeyGen / D-ID API** | 真人级，口型同步精准 | 按分钟计费，较贵 | 200-500ms（API 生成） | 低，调 API 即可 |
| **B: SadTalker** | 面部驱动，口型较准 | 免费，需 GPU（≥6GB 显存） | 本地推理 1-3s/秒视频 | 中 |
| **C: MuseTalk** | 面部+头部运动，口型精准 | 免费，需 GPU（≥8GB 显存） | 实时流式（优化后） | 中高 |
| **D: Live2D + TTS 口型同步** | 二次元风格，非真人 | 免费 | 实时 | 中，前端渲染 |
| **E: VRM + 动作捕捉** | 3D 虚拟人，可定制 | 免费 | 实时 | 高 |

#### 推荐路径

```
Phase 2a（快速出效果）
  → 方案 A：接 HeyGen/D-ID API，传文字/音频 → 返回数字人说话视频
  → 前端用 <video> 循环播放，面试流程代码零改动
  → 缺点：按分钟计费、API 延迟叠加

Phase 2b（降本 + 低延迟）
  → 方案 C：自建 MuseTalk
  → 以 TTS 音频 + 一张面试官照片 为输入，实时生成说话面部视频
  → 前端通过 WebRTC 推流播放，延迟 < 500ms
  → 需要一台带 GPU 的服务器
```

#### Phase 2 前端变更

Phase 1 的面试间 UI 不变（题目卡片、倒计时、对话流都复用），仅将：

- 静态头像 → `<video>` 播放数字人视频流（API 方案）或 WebRTC 流（自建方案）
- 波形动画 → 数字人嘴自然在动，不再需要

#### 面试录制

- [ ] Phase 1：录制音频 + 对话文字记录
- [ ] Phase 2：录制整个浏览器画面（数字人视频 + 对话叠加层），使用 MediaRecorder + Canvas 合成
- [ ] 回放页面：视频 + 时间轴标注（点击题目跳转到对应时刻）
